# Remote Control — управление серверами и генерация клиентских конфигов

Скрипт запускается на **вашем локальном компьютере**. Подключается к серверам по SSH, настраивает протоколы, генерирует клиентские конфиги для устройств, публикует подписки и управляет AmneziaWG-peers.

## Запуск

### Windows

Двойной клик по `remote-control.cmd` в корне проекта или из cmd:
```cmd
remote-control.cmd
```

Скрипт автоматически запускается через WSL.

### Linux / macOS

```bash
bash remote-control/remote-control-essence.sh
```

### Установленная версия (Linux / macOS)

```bash
curl -fsSL https://raw.githubusercontent.com/Morvex885/essence-setup/main/remote-control/install-remote-control.sh | sudo bash
remote-control-essence
```

### Зависимости

| Утилита | Где взять |
|---|---|
| `ssh`, `scp` | Встроены в Linux и macOS. На Windows — WSL |
| `openssl`, `base64` | Встроены |
| `jq` | `apt install jq` / `brew install jq` |

Отсутствующие зависимости скрипт пробует установить сам через `common/ensure-deps.sh`.

```
+------------------------------------------+
| Essence Remote Management                |
|------------------------------------------|
| 1) de-vps   1.2.3.4:22                  |
| 2) ru-vps   5.6.7.8:22                  |
|------------------------------------------|
| a) Добавить ноду                         |
| n) Переименовать ноду                    |
| t) Тег ноды                              |
| d) Удалить ноду                          |
|------------------------------------------|
| C) Клиенты                               |
| G) Группы                                |
|------------------------------------------|
| P) Подключения нод для групп             |
| W) AWG подключения                       |
|------------------------------------------|
| F) Сгенерировать конфиги                 |
| S) Подписки (Subscriptions)              |
|------------------------------------------|
| U) Обновить скрипт                       |
| L) Пароль скрипта                        |
| R) Удалить remote-control                |
| 0) Выход                                 |
+------------------------------------------+
```

- **Цифры** — выбор ноды для управления (SSH в меню Essence Setup на сервере)
- **Буквы** — локальные операции (клиенты, группы, генерация, подписки)

### Меню операций (после выбора ноды)

После выбора ноды скрипт подключается по SSH и открывает интерактивное меню Essence Setup на сервере. После выхода (пункт 0) возвращается к локальному меню ноды:

- **1) Открыть меню сервера** — (повторно) зайти в интерактивный Essence Setup
- **h) Настройка SSH ключа на ноде** — SSH Hardening (см. ниже)
- **u) Обновить скрипты на сервере** — `scp` актуальной версии на ноду

## Концепции

### Ноды

Нода — это VPS-сервер с установленным essence-setup. Хранит SSH-креды (пароль шифруется AES-256-CBC), имя, IP, порт, опциональный тег (`DE`, `RU` и т.п.). При выборе ноды открывается интерактивная SSH-сессия для настройки протоколов.

Переименование ноды каскадно обновляет ссылки в `.connections[]`, `.clients[].nodes[]`, `.subscription_host`. При удалении ноды скрипт предлагает удалить все компоненты essence на сервере.

### Группы

Группа — тип устройства. По умолчанию: **ROUTER**, **PC**, **MOBILE**.

Каждой группе назначается шаблон (по умолчанию `default.yaml`). Блоки шаблона, помеченные именем группы, попадают в конфиги клиентов этой группы (см. раздел [Шаблоны](#шаблоны)).

Список нод группы **производный** — высчитывается из `.connections[]`: группа считается активной на ноде, если в ней есть хоть один прокси этой ноды. Явного поля `.groups[].nodes` нет.

### Клиенты

Клиент — конкретное устройство (роутер, телефон, ноутбук). Привязан к группе. При создании автоматически получает:

- `vless_uuid` (генерируется `uuidgen`)
- `hy2_password` (генерируется `openssl rand -hex 16`)

Эти креды подставляются в сгенерированный конфиг и синхронизируются на ноды при генерации (`_sync_listeners_on_nodes`), чтобы каждый клиент имел свой идентификатор на сервере.

По умолчанию клиент наследует ноды от группы (`inherit_nodes_from_group: true`). Через **C) → e)** можно:

- **Назначить свои ноды** — отвязать от группы и указать явный `.clients[].nodes[]`
- **Настроить кастомные подключения** — per-nodes выбор, какие прокси использовать
- **Поменять группу** — с автосинком listeners на старых и новых нодах
- **Переименовать клиента** — с миграцией директории сгенерированных конфигов

### Подключения нод для групп (Connections)

Каждая нода может иметь несколько подключений: VLESS TCP/xHTTP/gRPC, Hysteria2, AmneziaWG, каскадные ноды. Не все из них нужны всем группам.

**P)** в главном меню:

1. На входе скрипт автоматически определяет, какие ноды ещё не синхронизированы, и предлагает дискавер
2. Обзор нод — цветом показано покрытие подключений по группам (зелёный/жёлтый/тусклый)
3. Выбрать ноду → скрипт по SSH читает `/etc/mihomo/client-config.txt` и выводит список подключений
4. **b) Назначить подключения группам** — для каждой группы отметить, какие прокси доступны

```
de-vps:
  VLESS TCP — ROUTER PC MOBILE
  VLESS xHTTP — PC MOBILE
  HY2 — ROUTER PC
  AWG (не назначено)
```

Stale-подключения (прокси, удалённые на сервере) автоматически вычищаются из `.connections[]` при каждом дискавере.

### AWG подключения (W)

Отдельное меню управления AmneziaWG-peers:

- Обзор нод с пометкой «нет AWG подключений», если на ноде нет прокси `AWG` в `.connections[]`
- По каждой ноде — список peers (используемые / orphan / missing) по сравнению с ожидаемым списком
- Создание недостающих peers батчем (один SSH на ноду)
- Удаление orphan-peers или удаление peer по номеру

Ожидаемые peers вычисляются из клиентов группы, которой на ноде назначено подключение `AWG`.

### Шаблоны

Шаблон — YAML-файл с полным конфигом mihomo, в котором блоки для разных групп обёрнуты маркерами.

#### Формат маркеров

```yaml
# --- ROUTER ---
redir-port: 1182
tproxy-port: 1181
# --- ROUTER ---

# --- PC/MOBILE ---
tun:
  enable: true
  stack: mixed
# --- PC/MOBILE ---
```

- `# --- ГРУППА ---` ... `# --- ГРУППА ---` — начало и конец блока
- `# --- ГРУППА1/ГРУППА2 ---` — блок активен для нескольких групп (разделитель `/`)
- При генерации: если группа клиента входит в список — содержимое остаётся, маркеры убираются. Если не входит — **весь блок удаляется**

#### Управление шаблонами

- Расположение: `remote-control/templates/*.yaml`
- Редактирование: **F) → t)** — выбор шаблона или создание нового
- Назначение группе: **G) → t)** — выбрать группу и шаблон

## SSH Hardening

Пункт **h)** в меню операций ноды. Выполняет за один шаг:

1. Генерирует SSH-ключ (ed25519) если ещё нет, копирует на сервер
2. Отключает вход по паролю (`PasswordAuthentication no`)
3. Меняет SSH-порт на случайный (49152-65535)
4. Обновляет UFW/firewalld, чтобы пропускать новый порт

Каждый шаг проверяется перед переходом к следующему (`sshd -t`, тестовое соединение). При ошибке на любом этапе — автоматический откат.

## Генерация конфигов

**F)** в главном меню:

- **g)** По группе — обновить конфиги всех клиентов группы
- **a)** Всем клиентам — полный пересчёт по всем группам
- **t)** Редактировать шаблон

### Что происходит при генерации

1. Берётся шаблон группы
2. Удаляются блоки, не относящиеся к группе клиента
3. Для каждой ноды клиента скрипт забирает proxy-конфиги с сервера (кешируются на время сессии)
4. Прокси фильтруются по подключениям группы + per-client кредам (UUID, hy2 password)
5. Проверка на дубликаты proxy-имён
6. Готовый `config.yaml` сохраняется в `remote-control/.remote-data/generated/<group>/<client>/`
7. **Listener sync** — один SSH на каждую затронутую ноду: серверные листенеры обновляются, чтобы принимать per-client UUID/пароль
8. **Subscription refresh** — если у клиентов есть опубликованные подписки, скрипт предлагает их обновить

## Подписки (Subscriptions)

**S)** в главном меню. Подписки хостятся через nginx на одной из нод (хост-нода) — клиент получает YAML-конфиг по HTTPS-ссылке вида `https://sub.example.com:<port>/sub/<token>`.

### Пункты меню

| Пункт | Действие |
|---|---|
| **1) Выбрать host-ноду** | Назначить ноду, на которой установлен `s)` Subscription hosting. Скрипт читает `subscription.conf` и кеширует `base_url`, `sub_dir`, `nginx_group` |
| **2) Опубликовать подписку** | Сгенерировать токен (`openssl rand -hex 32`), загрузить `<token>.yaml` и per-token nginx snippet, проверить HTTP 200 |
| **3) Обновить все подписки** | Пакетный апдейт. Клиентам без токена выдаётся новый. Orphan-файлы (токены удалённых клиентов) подчищаются |
| **4) Показать подписку** | URL, TTL, expires_at, resolved-headers |
| **5) Список подписок** | Таблица по всем клиентам: токен (скрыт), дата, статус |
| **6) Сменить токен** | Новый токен, удаление старого yaml/snippet |
| **7) Отозвать подписку** | Удаление yaml, snippet и токена из config.json. Заголовки клиента сохраняются |
| **8) Миграция на другую ноду** | Переезд всех подписок со старой ноды на новую |
| **h) Header клиента** | Установить / удалить HTTP-заголовок для конкретного клиента |
| **g) Header группы** | Установить / удалить заголовок, наследуемый всеми клиентами группы |
| **r) Показать resolved headers** | Итоговые заголовки с указанием источника (default / group / client) |

### Наследование заголовков

```
default  (из .subscription_default_headers)
   ↓ переопределяется
group    (из .groups[].subscription_headers)
   ↓ переопределяется
client   (из .clients[].subscription.headers)
```

По умолчанию устанавливаются:

- `Content-Disposition: attachment; filename="config.yaml"` — имя файла при скачивании
- `profile-update-interval: 24` — интервал обновления у клиентов (mihomo / Clash.Meta)

Заголовки рендерятся в per-token nginx snippet с автоматическим квотированием (одинарные кавычки по умолчанию, двойные с escape при наличии одинарной кавычки в значении).

## Пошаговый гайд

### 1. Добавить ноду

**a)** → ввести IP, порт, логин, пароль → скрипт проверит SSH → предложит сохранить.

### 2. Настроить сервер

Выбрать ноду по номеру → настроить протоколы (VLESS, Hysteria2, AmneziaWG, WARP).

### 3. (Опционально) SSH Hardening

В меню ноды: **h)** → скрипт настроит ключ, отключит пароль, сменит порт.

### 4. Назначить подключения группам

**P)** → выбрать ноду → **b)** → отметить подключения для каждой группы.

### 5. Добавить клиентов

**C)** → **a)** → ввести имя → выбрать группу. Credentials сгенерируются автоматически.

### 6. Сгенерировать конфиги

**F)** → **a)** (всем) или **g)** (по группе). Готовые конфиги в `remote-control/.remote-data/generated/`.

### 7. (Опционально) Опубликовать подписки

На host-ноде: **8) Клиентский конфиг → s) Subscription hosting → 1) Установить**.

Локально: **S) → 1) Выбрать host-ноду → 3) Обновить все подписки**.

## Структура config.json

```jsonc
{
  "nodes": [{                          // Список серверов (VPS)
    "name": "de-vps",                  // Уникальное имя ноды
    "ip": "1.2.3.4",                   // IP или домен
    "port": 22,                        // SSH-порт
    "user": "root",                    // SSH-пользователь
    "auth": "key",                     // "key" или "password"
    "pass": "U2FsdGVk...",             // AES-256-CBC (только при auth=password)
    "tag": "DE"                        // Опциональный тег (отображается в меню)
  }],
  "groups": [{                         // Группы устройств
    "name": "ROUTER",                  // Имя группы
    "template": "default.yaml",        // Шаблон конфига
    "subscription_headers": []         // HTTP-заголовки, наследуемые клиентами группы
  }],
  "clients": [{                        // Устройства
    "name": "my-phone",            // Уникальное имя
    "group": "MOBILE",                 // Группа
    "inherit_nodes_from_group": true,  // true — ноды от группы, false — свои
    "nodes": [],                       // Свой список нод (при inherit=false)
    "connections": [],                 // Кастомные подключения (при inherit=false)
    "credentials": {                   // Per-client креды (генерируются автоматически)
      "vless_uuid": "...",
      "hy2_password": "..."
    },
    "subscription": {                  // Опубликованная подписка
      "token": "...",                  // 64-hex токен
      "created_at": "2026-04-20T12:00:00Z",
      "expires_at": null,              // null = без TTL
      "headers": []                    // Per-client HTTP-заголовки
    }
  }],
  "connections": [{                    // Какие прокси ноды доступны каким группам
    "node": "de-vps",
    "groups": [
      { "name": "ROUTER", "proxies": ["VLESS TCP", "HY2"] },
      { "name": "MOBILE", "proxies": ["VLESS TCP"] }
    ]
  }],
  "subscription_host": {               // Нода-хост подписок
    "node": "de-vps",
    "base_url": "https://sub.example.com:2096",
    "sub_dir": "/var/lib/essence-sub",
    "nginx_group": "www-data"
  },
  "subscription_default_headers": [    // Заголовки по умолчанию
    { "name": "Content-Disposition", "value": "attachment; filename=\"config.yaml\"" },
    { "name": "profile-update-interval", "value": "24" }
  ]
}
```

## Хранение данных

| Режим | Директория |
|-------|-----------|
| Dev (из папки проекта) | `remote-control/.remote-data/` |
| Installed (Linux/macOS) | `~/.config/remote-control-essence/` |

```
remote-control/.remote-data/
+-- config.json         — ноды, группы, клиенты, подключения, подписки
+-- .auth               — хеш пароля на скрипт (опционально, 600)
+-- generated/
    +-- <group>/
        +-- <client>/
            +-- config.yaml — сгенерированный конфиг
```

## Пароль на скрипт

**L)** — установить / сменить / убрать пароль на запуск remote-control. Хеш (SHA512 с солью) хранится в `.auth` с правами 600. После 3 неудачных попыток скрипт выходит.
